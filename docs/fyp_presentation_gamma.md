# The Problem I Noticed

Before starting this project, I tried using different AI tools to look up nutrition info for the same food.

**The results varied by as much as 2x** — same food, completely different calorie and macro numbers.

This made me realize: the problem isn't that AI isn't powerful enough. The problem is that AI generates nutrition answers **without reliable data sources**.

---

# Two Core Technical Problems

## Problem 1: Nutrition Hallucination

- LLMs generate **plausible-sounding but inaccurate** nutrition information
- No traceable data source behind the answers
- High-risk in nutrition scenarios — wrong info can mislead users

## Problem 2: Inaccurate Portion Estimation

- Single-photo estimation is unreliable
- Affected by: shooting angle, distance, lighting, food occlusion, container shape, dish complexity
- No single method works well in all conditions

---

# Why This Matters

This project is **not** just about automating food logging.

It's about researching two questions:

1. **Can RAG reduce nutrition hallucination** when LLMs answer nutrition questions?
2. **How do different portion estimation methods compare** in real-world conditions?

As an AI & EdTech project: a tool that gives unreliable information **cannot be an effective learning tool** — this recognition elevated RAG from a technical choice to a **design principle**.

---

# Research Questions

**RQ1:** Does integrating Retrieval-Augmented Generation (RAG) with local authoritative nutrition data reduce hallucination in LLM-generated nutrition answers?

**RQ2:** How do different food portion estimation methods (ARCore measurement, EXIF-based calibration, pure LLM vision) compare in accuracy, stability, and usability?

# Pipeline 1: Food Logging Line

### 3-Step RAG Pipeline

```
Step 1: Identify
Photo + Camera Metadata → Gemini → Food Name(s)

Step 2: Retrieve
Food Name(s) → CFS Database
  ├── Chinese text search (ILIKE)
  └── pgvector semantic search (text-embedding-3-small, 768 dims)
→ Matched CFS nutrition records

Step 3: Analyze
Photo + Camera Context + CFS Data → Gemini → Final Nutrition Estimate
```

User can **edit all values** before saving to dashboard.

# Methodology: RAG for Nutrition Hallucination

### Why RAG instead of fine-tuning?

Based on Lewis et al. (2020) — the original RAG paper for knowledge-intensive NLP tasks:

| Approach | Knowledge Location | Updateability | Traceability |
|----------|--------------------|---------------|--------------|
| Fine-tuning | Stored in model parameters | Requires retraining | Cannot trace source |
| RAG | External database | Update data independently | Source clearly identified |

Ji et al. (2023) hallucination survey confirms: **parametric knowledge has inherent uncertainty in factual recall** → retrieval-grounded generation shifts the problem from "model memory uncertainty" to "retrieval coverage" — a more controllable dimension.

---

# Methodology: RAG Implementation Details

### Hybrid Retrieval Strategy

| Method | How It Works | When It Helps |
|--------|-------------|---------------|
| Chinese text search (ILIKE) | Exact/partial match on Chinese food names | Common foods with standard names |
| pgvector semantic search | text-embedding-3-small (768 dims) cosine similarity | Foods with varied naming conventions |

### Knowledge Source

- **Hong Kong Centre for Food Safety (CFS)** nutrition composition database
- Stored in Supabase PostgreSQL with pgvector extension
- ~1,000+ food items with per-100g nutrition data

# Methodology: Multiple Portion Estimation Methods

### Four approaches compared:

| Method | Geometric Context | Automation | Hardware Requirement |
|--------|-------------------|------------|----------------------|
| **ARCore Manual** | 3D bounding box via depth sensing | Low (user places points) | ARCore-capable device |
| **ARCore + SAM** | Mask-guided depth sampling | High (automatic segmentation) | ARCore + on-device SAM |
| **EXIF + LLM** | FOV & pixel-to-cm from 35mm equiv. focal length | High (automatic from EXIF) | Any camera with EXIF |
| **Pure LLM** | None (visual estimation only) | Highest | Any camera |

The goal is **not** to prove one method is always best, but to understand **trade-offs and suitable conditions**.

---

# Methodology: EXIF FOV Calibration

### How camera metadata becomes geometric context

```
Input:  EXIF 35mm equivalent focal length
        Sensor dimensions (derived from standard)
        Image resolution

Formula:
  FOV_horizontal = 2 × arctan(sensor_width / (2 × focal_length_35mm))
  degrees_per_pixel = FOV_horizontal / image_width
  cm_per_pixel = 2 × distance × tan(degrees_per_pixel / 2)

Output: OPTICS block injected into Gemini prompt
  → "At the estimated shooting distance, each pixel ≈ X cm"
```

This gives the LLM **quantitative spatial reasoning basis** instead of pure visual guessing.

# Methodology: Human-in-the-Loop Design

### Why not fully automatic?

Referenced from HCI literature:

- **Amershi et al. (2019)** — CHI 2019 Guidelines for Human-AI Interaction: AI systems should let users **correct errors**, **understand system reasoning**, and receive **clear uncertainty signals**
- **Shneiderman (2020)** — Human-Centered AI framework: high-quality AI design maintains **both** high human control **and** high automation

### Applied in SmartDiet AI:

- Manual nutrition editing on **both** camera result page and dashboard meal cards
- RAG source badge ("CFS Official" vs "AI Estimate") on every result
- Expandable pipeline detail showing each analysis step

# Integration 2: Camera Optics × AI

**From:** Lo et al. (2020) reviewed image-based food volume estimation methods using camera calibration (intrinsic parameters, reference objects) for geometric context.

**Applied to:** Extracting 35mm equivalent focal length from EXIF → computing pixel-to-cm formula → injecting as OPTICS block into Gemini prompt.

**Why cross-domain:**
- Pure AI background → would not think to derive physical dimensions from EXIF metadata
- Pure photography/optics background → would not think to convert it into LLM prompt context

This is a **mobile-context extension** of camera calibration for food estimation.

# Integration 3: HCI × Computer Vision

**From:** Amershi et al. (2019) — 18 guidelines for human-AI interaction; Shneiderman (2020) — HCAI framework balancing human control and automation.

**Applied to:** Designing manual editing, source transparency, and pipeline visibility as **architectural requirements**, not optional features.

**Why cross-domain:**
- HCI principles alone → don't know that food portion estimation is inherently high-uncertainty
- CV/AI knowledge alone → might default to pursuing full automation

# Key Feature 2: Multi-Method Portion Estimation

### ARCore 3D Bounding Box
- Uses depth sensing and plane detection
- User-guided or SAM mask-guided depth sampling
- Outputs physical dimensions (L × W × H in cm)

### EXIF FOV Calibration
- Extracts 35mm equivalent focal length
- Computes horizontal FOV and pixel-to-cm ratio
- Injected as OPTICS block in prompt

### Pure LLM Visual Estimation
- No geometric context provided
- Model relies entirely on visual judgment
- Fallback when no metadata available

# Limitations and Trade-offs

### RAG Limitations:
- Retrieval quality depends on **CFS database coverage**
- Chinese food naming variations can affect text search accuracy
- RAG reduces but **does not eliminate** hallucination risk

### Portion Estimation Limitations:
- All methods affected by **occlusion, lighting, mixed dishes, container shape**
- ARCore requires compatible hardware and stable surface detection
- EXIF availability and accuracy varies across devices

### Deliberate Trade-off:
System positioned as **AI-assisted, evidence-aware, human-correctable** — not fully automatic. Reliability and correctability > surface-level automation.
